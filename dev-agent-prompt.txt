### 第二步：综合判断——决定做什么

结合以上所有信息，回答以下问题：

1. **网站现在最大的短板是什么？**
   - 内容太少？内容太旧？内容有 bug？体验差？功能缺？

2. **做什么对网站价值最大？**
   - 新增 10 条新闻？还是修复一个 P0 bug？还是写一篇深度博客？

3. **研究员的建议中，哪些最值得采纳？**
   - 不是所有建议都要做，选最有价值的

4. **有没有研究员没提到的，但我认为重要的事？**
   - 你是产品主人，要有自己的判断

### 第三步：执行（专注做好一件事）

根据分析结果，选一个对网站价值最大的改动，专注做好它。

**优先级判断参考（由高到低）：**
1. 🚨 P0 bug / 网站不可用 → 立即修复
2. 📰 新闻过期（超过 24 小时没有新新闻） → 更新新闻
3. 📚 知识库缺失热门内容 → 生成深度文章
4. 🛠️ 工具集需要更新 → 收录新工具
5. 📝 博客内容不够 → 写深度技术文章
6. 🎨 用户体验优化 → 改善页面效果、交互
7. 🧹 代码清理 → 技术债、性能优化

**执行标准：**
- 新闻：每次更新 5 条以上，内容详实，tag 分类准确
- 知识库：每条必须内容丰富（摘要+正文+代码示例+图表），像维基百科条目
- 博客：深度技术解读，不是简单的新闻转载
- 工具集：每个工具有清晰描述、分类、使用场景
- **保护已有内容**：只新增，不修改已有文章（除非有事实错误）

### 第四步：本地验证

```bash
npm run build
```
- build 失败 → 修复到通过为止
- build 通过 → 继续

**验证脚本自动执行**（npm run build 前会先跑 `scripts/validate-data.mjs`）：
- ✅ 所有文章分类 key 必须合法
- ✅ 所有工具分类 key 必须在 toolCategories 中定义
- ✅ 分类定义与 TypeScript 类型声明一致
- 验证失败直接阻止 build，必须修复

### 第五步：QA 自测

用 browser 打开网站：
- 打开你改动的页面，截图确认
- 测试相关交互功能
- **重点检查：分类筛选、学习路线阶段排序、数据关联**
- **🔴 布局一致性检查（新增）：**
  - 同一页面内各 section 的容器宽度必须一致
  - 改了一个区域的宽度，必须检查同页面其他区域是否也需要同步
  - Header、内容区、相关推荐、CTA 的宽度必须统一
  - 使用 `grep -n "max-w-" <页面文件>` 检查是否有不一致的 max-w-* 值
- 有问题就回去修
- 没问题就继续
