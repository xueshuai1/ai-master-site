- 完整配色和语法规范见下方「Mermaid 图表配色和类型」

### 1. Mermaid 图表配色和类型（2026-04-28 事故后增强）
- ❌ **禁止使用 `style X fill:#颜色` 语法** — 这是历史事故根因！`style fill` 会覆盖 mermaid 默认样式，导致节点文字不可见、图表变黑块/空白块（aieng-021 事故）
- ✅ **必须使用 `classDef` + `class` 语法** — 如下方示例，这是唯一合法的节点着色方式（2026-04-28 事故后增强）

**🔴 语法铁律：**
- ❌ **禁止使用 `style X fill:...` 语法** — 会导致渲染为空白块/黑块（aieng-021 事故）
- ✅ **必须使用 `classDef` + `class` 语法** — 详见下方示例
- ❌ **禁止使用浅色**（400-500 色阶）和 Material Design 浅色：`#ef4444` `#f59e0b` `#10b981` `#6366f1` `#8b5cf6` `#ec4899` `#f3e5f5` `#e1f5fe` `#c8e6c9` `#fff3e0` `#8AB4F8` `#E8EAED` `#F28B82`
- ✅ **必须使用深色**（800-900 色阶）+ 白字 `color:#fff`：`#991b1b` `#92400e` `#064e3b` `#3730a3` `#581c87` `#1e3a5f`
- ✅ **classDef 填充色亮度不得超过 0.4**（亮度 = 0.2126*R + 0.7152*G + 0.0722*B），否则文字不可读
- 完整配色表：`docs/MERMAID-COLORS.md`

**🔴 连线语法铁律（2026-04-29 blog-084 事故新增）：**
- ❌ **禁止使用 `.- >`（中间带空格）** — 会导致连线断裂、节点渲染为空白块
- ✅ **连线箭头必须是 `->` 连续字符**，如 `A --> B`、`A -.标注.-> B`
- ❌ **禁止** `.- >`、`. - >`、`. -  >` 等任何带空格变体
- ✅ **虚线标注箭头格式**：`A -.text.-> B`（注意 `.` 在 text 前后都要有）

**🔴 % 符号铁律（2026-04-29 blog-084 控制台报错事故新增）：**
- ❌ **禁止在 mermaid 块中使用 `%` 字符** — mermaid 中 `%` 是注释标记，会导致整行解析错误
- ✅ **必须使用全角 `％`** 代替半角 `%`，如 `15-25％` 而非 `15-25%`
- ❌ **`\%` 转义无效** — JS 模板字面量不识别 `\%`，会静默丢掉反斜杠，mermaid 仍然看到 `%`

**✅ 正确示例（必须用 classDef）：**
```
graph TB
    A[节点1] --> B[节点2]
    classDef important fill:#991b1b,stroke:#b91c1c,color:#fff;
    class A important;
```

**❌ 错误示例（禁止！）：**
```
graph TB
    A[节点1] --> B[节点2]
    style A fill:#991b1b  ← 禁止！必须用 classDef
    style B fill:#1e3a5f  ← 禁止！这是 aieng-021 事故根因

**✅ 合法的 Mermaid 图表类型（只能用以下类型）：**
- `graph TD` / `graph LR` / `graph BT` / `graph RL`（流程图/架构图）— **最推荐**
- `sequenceDiagram`（时序图）
- `pie`（饼图）
- `stateDiagram-v2`（状态图）
- `classDiagram`（类图）
- `erDiagram`（实体关系图）
- `journey`（用户旅程图）
- `quadrantChart`（象限图）
- `mindmap`（思维导图）

**⚠️ 慎用类型：**
- `gantt`（甘特图）— 中文标签过长会导致文字重叠、日期截断，**不推荐用于中文内容**。如需展示时间线/路线图，改用 `graph LR` + subgraph 分组

**❌ 非法的 Mermaid 图表类型（禁止使用）：**
- `tree-beta` — 不是合法类型，会渲染为空白块
- `flowchart` 后必须跟方向：`flowchart TD`，不能单独用 `flowchart`
- 任何不在上述合法列表中的类型

### 2. 代码块格式（最高频出错！）
- 🔴 **代码块是可选的，但 Mermaid 图表是必须的。** 文章可以没有代码，但不能没有图表。
- ❌ **禁止**在 body 中写 `\`\`\`lang ... \`\`\` 代码块 — MarkdownBody 不会渲染语法高亮
- ✅ **必须**把代码放在 `code: [{ lang, code, title }]` 数组中
- body 中用**描述性文字**指向代码，如「（见下方代码：LessHumanAgent 完整实现）」
- **每个 code 块必须有 `title` 字段**，与 body 中的引用文字对应

**正确示例（整个 section）：**
```
{
  title: "五、Python 实战",
  body: `下面我们用 Python 实现。

### 实现 1：核心 Agent

（见下方代码：LessHumanAgent 完整实现）

### 实现 2：约束验证

（见下方代码：约束验证器实现）`,
  code: [
    {
      lang: "python",
      title: "LessHumanAgent 完整实现",
      code: `import json
import time
...
# 完整代码放这里`,
    },
    {
      lang: "python",
      title: "约束验证器实现",
      code: `from enum import Enum
...
# 完整代码放这里`,
    },
  ],
}
```

**错误示例（不要这样写！）：**
```
// ❌ 错误1：代码嵌在 body 里
body: `文字...
\`\`\`python
import json
\`\`\`
...文字`,

// ❌ 错误2：code 块没有 title
code: [{
  lang: "python",
  code: `import json...`,
}],

// ❌ 错误3：body 用「见代码块1」这种无意义引用
body: `...（见代码块 1）...`,
```

- 来源必须是**技术方向的权威媒体/机构**（arXiv、OpenAI Blog、Anthropic Blog、Google AI Blog、DeepMind Blog、Simon Willison、TechCrunch、The Verge、MIT Technology Review 等）
- ❌ **禁止**把「本站博客更新」「本站知识库新增」「本站工具收录」当新闻写
- ❌ **禁止** source 字段写 "AI Master 博客"、"AI Master 知识库"、"AI Master 自主创作" 等站内来源
- ❌ **禁止**非技术方向的娱乐/八卦类 AI 新闻
- ❌ **禁止**使用自媒体/营销号作为新闻来源
- ✅ 如果写了新博客/知识库文章，可以在 news.ts 中用**外部视角**写一条（如："Anthropic 发布 Claude Opus 4.7" 而不是 "本站发布 Claude Opus 4.7 解读"）
- ✅ 新闻源必须**广泛**，不要一天只盯一个来源
- 🔴 **同一事件多渠道综合**：同一个事件如果被多个来源报道，**合并为一条新闻**，但在内容中列出各方的不同观点（TechCrunch 关注定价、The Verge 关注功能、Simon Willison 关注技术细节等）
- 🔴 **采集量要大**：不封顶，有价值的全收录，**不要为了省事而缩减新闻条数**

**你做决定，不需要等任何人同意。**

**参考方向（但不限于此）：**
- 新闻过期 → 抓取最新 AI 动态，更新 news.ts
- 知识库缺失 → 生成深度文章（摘要+正文+代码示例+图表）
- 有 bug → 修复 QA 报告中的问题
- 工具集过时 → 更新 tools/page.tsx
- 用户体验差 → 增加互动、动画、搜索优化
- 页面单调 → 丰富视觉元素（图标、插图、动画）
- 你觉得需要什么 → 就去加什么

**🔴 博客与知识库的定位差异（重要！）：**

> **「技术博客一定要比知识库更具有专业水准，更深度。」**

**✅ 可以是同一话题，但必须方向不同：**

| 维度 | 知识库 | 博客 |
|------|--------|------|
| **定位** | 技术百科/教程/指南 | 趋势分析+深度解读+专业观点 |
| **写法** | 百科式，像维基百科 | 杂志式，像 Medium 深度文章 |
| **深度** | 教程/指南级别 | **专业研究级别，比知识库更深一个层次** |
| **角度** | 「Voice AI 全景指南：技术原理+实战教程」 | 「Voice AI 爆发：三大项目周增 15K 星背后的行业趋势」 |
| **侧重点** | 系统性知识、API 文档、代码示例 | 深入技术拆解、数据对比、趋势预判 |

**示例对比：**
- 知识库写「Transformer 架构详解」→ 把 Self-Attention、Multi-Head、Position Encoding 讲清楚
- 博客写「Transformer 的注意力瓶颈：为什么 Linear Attention 才是未来」→ **深入分析** O(n²) 复杂度问题，对比 5 种 Linear Attention 方案，给出实验数据和未来预判

**🔴 铁律：话题去重（2026-04-22 用户指示）**

> **「如果要写同样话题，优先在之前文档里做更新内容处理。除非两篇文章的方向相差比较远，才可以写两篇不一样的。」**

**写前必查：**
```bash
# 检查知识库已有文章
grep 'title:' src/data/articles/*.ts | grep -i "关键词"
# 检查博客已有文章
grep 'title:' src/data/blogs/blog-*.ts | grep -i "关键词"
```

**决策：**
- 发现已有同话题文章 → 想写的角度和已有文章方向相差远吗？
  - 是（角度完全不同）→ ✅ 写新文章，注明角度差异
  - 否（相近）→ ❌ 不要另写一篇，**更新旧文**

**什么叫「方向相差远」：**
- ✅ 「Transformer 架构详解」vs「Linear Attention vs Self-Attention：O(n²) 瓶颈分析」→ 原理介绍 vs 深度对比
- ✅ 「Claude Opus 4.7 技术文档」vs「Opus 4.7 Tokenizer 变革：开发者成本分析」→ 技术文档 vs 成本实战
- ❌ 「RAG 技术全景」vs「RAG 技术指南」→ 同一件事换标题
- ❌ 「Voice AI 全景指南」vs「Voice AI 技术全景」→ 标题不同但内容重复

**最终判断标准：**
> 问自己：「读者先看了已有文章再看新文章，觉得有收获还是重复？」
> - 有收获（角度完全不同）→ ✅ 新文章
> - 重复（同一件事换说法）→ ❌ 更新旧文
> - 不确定 → 默认更新旧文

**更新旧文的方式：**
- 不删除原有内容
- 在相关章节末尾追加「**更新于 YYYY-MM-DD**：...」
- 新增「**最新动态**」章节
- commit message 注明更新了哪些文章

**博客内容标准：**
- 博客使用 `ArticleSection[]` 结构化格式（不是 markdown 字符串）
- 每篇至少 2 个 Mermaid 图（架构图/流程图/对比图/时间线）
- 每篇至少 2 个 Python 代码块（可运行的示例代码）
- 至少 1 个对比表格
- 章节按 `title/body/code/mermaid/table/list/tip/warning` 拆分
- 不能是几句话的摘要，要有深度（3000+ 字）
- 来源必须是国外权威网站（arXiv、OpenAI Blog、Anthropic、TechCrunch 等）

**知识库内容标准（重点！）：**
- 每条知识必须内容丰富、有深度，不能只有几行字
- 必须包含：摘要、分章节正文、代码示例（带语法高亮）、架构图/流程图/对比表格
- 来源必须是国外权威网站（OpenAI、DeepMind、Anthropic、MIT Tech Review、TechCrunch、arXiv 等）
- 像维基百科条目一样详实，不是简单的摘要
- 每次运行至少要新增或大幅改进 1-2 条知识条目

## 📝 老文章更新机制

**知识库文章不仅新增，还要定期更新已有内容。**

**触发条件：**
- 内容研究报告中注明"建议更新文章 xxx"（如某主题有新进展）
- 某篇文章内容过时（如工具版本更新、方法被替代）
- 热点新闻涉及已有知识库文章（如 LoRA 有重大更新）

**更新方式：**
- 不删除原有内容，在原有章节末尾追加"更新于 YYYY-MM-DD：..."
- 新增"最新动态"章节，记录该主题的最新进展
- 更新文章的 tags、summary（如有变化）
- 不修改文章 ID 和基础结构

**更新频率：**
- 热门主题（LLM、Agent、RAG 等）：每月检查一次
- 成熟主题（数学基础、经典 ML 算法）：每季度检查一次

**注意：**
- 更新时保持文章原有的风格和结构
- 如果有事实错误（如版本号、API 变更），直接修正
- 每次更新要在 commit message 注明更新了哪些文章
